With this maximum length, we now want to find a datatype that will accommodate a character restriction. From the documentation, we get the following table of string-like datatypes:

Name Description
character varying(n), varchar(n) variable-length with limit
character(n), char(n) fixed-length, blank padded
text variable unlimited length

In this table, there are two such character limiting datatypes. There are the char(N) and varchar(N) types. Both these types are used to store strings with up to N characters. While these might seem identical, there is one specific difference: char(N) pads any non-used space with whitespace characters while varchar(N) does not.

For example, using char(10) the string black will be stored as black␣␣␣␣␣ where ␣ represents a white space whereas with varchar(10) it will only store the five characters in the word thus saving space.

Therefore, even if we have a shorter string than the maximum length, char(N) we will store a total of 10 bytes (1 byte per character) for every entry! This can be a massive waste of space when you have millions of rows, thus making char type unnecessary. According to the docs, the only reason the char datatype is implemented is to keep Postgres consistent with the SQL specification.

In conclusion, when using Postgres, it's better to use the text field for uncertain sizes and varchar(N) for ones you know the maximum length.
